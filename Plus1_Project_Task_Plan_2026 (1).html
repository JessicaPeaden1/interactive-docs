How to Use This Task Plan Template
Two ways to populate this document — pick the one that fits your workflow. Use after the Project Roadmap is approved and the project is moving forward.
🕐 When to Use This Template

This task plan is the execution layer of the project — it lives below the roadmap and above daily tickets. Use it after the project roadmap is approved and signed, to assign and track every unit of work through to client sign-off. Every task maps to a phase and milestone from the approved roadmap. This applies to any Plus 1 engagement — network, infrastructure, hardware, M365, VoIP, security, AI/automation, or mixed.

🔧 Set Your Project Type First

Use the Project Type selector bar directly below the header to show only the tabs relevant to your engagement. Each preset hides irrelevant groups:

🔧
MSP / Infrastructure
Core + Scope + Technical + MSP + Handoff. Best for network, hardware, server, and on-site projects.
🤖
AI / Automation
Core + Scope + Technical + AI/Automation + Handoff. Best for n8n, Make, API, and prompt engineering projects.
📦
Small / Contained
Core + Scope + Technical + Handoff only. Best for scoped, single-phase, or low-complexity projects.
⚡
Full (All 40 Tabs)
All 40 tabs visible. Best for large, complex, or multi-phase projects that span both MSP and AI workstreams.

⚙️ How the Data Works — Script Block Guide

All dynamic content — task cards, phase summaries, blocker cards, risk register, decisions log, ClickUp import table — renders automatically from five JavaScript arrays at the top of the <script> block. You never touch HTML — only the arrays.

Five data arrays to populate:
const PHASES = phase structure + criteria
const TASKS = all project tasks (main array)
const BLOCKERS = active blockers + open items
const DECISIONS = confirmed decisions log
const RISKS = risk register entries
Each TASKS entry must include:
id, title, phase, milestone
status, owner, estimate, estimateMs
startDate, dueDate, onSite, changeWindow
description, dependencies
checklist[ ], dod, risk, metadata{ }
