Project Summary

PhaseAlpha is building a SaaS product that automates HVAC equipment schedule extraction from PDF building plan sets, replacing the manual contractor transcription process that is slow, error-prone, and puts all accuracy risk on the contractor. Plus1 Technology is engaged to design, build, and validate the AI extraction pipeline that forms the technical core of this product — specifically Layers 3–7 of a seven-layer architecture (table detection, crop orchestration, extraction, normalization, validation). PhaseAlpha handles document ingestion and page classification upstream (Layers 1–2) and owns the Review UI downstream.

The core finding from document analysis: this is a detection + rendering + extraction + trust problem, not simply an OCR problem. A naïve single-pass approach produces unreliable results in production. The architecture is fail-safe by design — the system returns null for unreadable values rather than inferring or guessing, because bad data creates downstream liability for HVAC contractors.

The execution strategy is a two-phase build: an n8n pilot (Phase 0) validates the extraction logic quickly and cheaply before any .NET development commitment, then proven logic migrates into PhaseAlpha's .NET platform for production. Fixed-bid engagement. Quote signed March 28, 2026. Kickoff target: ~April 2, 2026.

Business Goals
⚡
Eliminate manual transcription

HVAC contractors currently transcribe all schedule data by hand — a process that takes hours per project and puts 100% of accuracy risk on the contractor.

💰
Reduce contractor liability exposure

Fabricated or inferred data downstream of this pipeline creates real pricing and configuration liability. The system's null-first policy directly reduces this risk for PhaseAlpha's customers.

🚀
Faster time-to-quote

Once extraction is reliable, contractors can produce equipment takeoffs in minutes rather than hours, dramatically increasing the volume of projects PhaseAlpha's platform can process per user.

📦
PhaseAlpha owns all IP

All code, workflows, prompts, and data models built in this engagement are owned entirely by PhaseAlpha. No vendor lock-in — built on independent APIs that can be swapped at any time.

Technical Goals
🔍
Two-pass table detection

Detect table regions at page scale (150 DPI), then crop and re-render at 300–500 DPI per table. VAV box confidence: 0.76 → 0.91 after cropping in documented testing.

🤖
LlamaParse + Claude extraction pipeline

LlamaParse as primary detector (Option A) with Docling fallback (Option B). Claude Sonnet/Opus for vision extraction. OpenAI Structured Outputs for schema normalization.

🗺
Canonical data model + synonym mapping

Seven canonical fields mapped from hundreds of header variants across engineering firms. Column header mapping agent resolves novel variants gracefully.

🛡
Validation and anti-hallucination layer

Eight failure codes (VALUE_ILLEGIBLE, FABRICATED_COMPLETENESS, DUPLICATE_TAG, etc.) with defined routing actions. Cross-model verification for worst-case fabrication detection.

Scope Boundaries

What Plus1 builds, what PhaseAlpha owns, and what is deferred to post-pilot.

✅
Scope confirmed from PhaseAlpha (Joe Lewis, March 28, 2026)
PhaseAlpha handles page splitting and routing upstream and delivers pre-split schedule pages as PDF. PhaseAlpha owns the Review UI. Reference files will be cleaned up and provided. Quote signed. These scope decisions materially reduced the estimate by ~4–6 days vs. original scope.
✅ In Scope — Plus1 Builds
🔍
Layer 3 — Table Detection

LlamaParse primary detector identifying every distinct equipment schedule table on each pre-split schedule page PDF, returning bounding boxes and equipment type labels. Docling fallback wired in if Option B selected.

✂️
Layer 4 — Rendering & Crop Orchestration

Two-pass pipeline: 150 DPI full-page scan for table location, then 300–500 DPI crop per table based on density. Full-width horizontal split-and-stitch for 40+ column AHU/RTU schedules. PyMuPDF (fitz) implementation.

🤖
Layer 5 — Extraction

Claude Sonnet/Opus vision model processing cropped table images. Returns raw cell-level content with null for unreadable values. Operates on crops only — never full-page images.

🗺
Layer 6a — Column Header Mapping Agent

AI agent with synonym knowledge base resolving novel header variants to canonical field names. Pre-populated with known variants from Bluestone, MODUS, and Shive Hattery analysis.

📐
Layer 6b — Schema Normalization

OpenAI Structured Outputs enforcing canonical equipment schema at the model response layer. Unit conversion, terminology normalization, remarks concatenation.

🛡
Layer 7 — Validation, Anti-Hallucination & Trust

Eight failure codes with routing actions. Cross-model verification for FABRICATED_COMPLETENESS detection. Confidence scoring and review queue routing rules.

🔄
n8n Pilot Orchestration (Phase 0)

Full pipeline wired in n8n for rapid iteration and validation. Self-hosted recommended. All 10 automation nodes (A1–A10) from intake through learning loop.

📋
API Payload Contract

Defined JSON handoff contract for PhaseAlpha's Review UI team — normalized fields, per-field confidence scores, failure codes, flagged cell references, crop image URL, source traceability. Required deliverable before PhaseAlpha UI development begins.

📊
Canonical Data Model

Document-level metadata, schedule-level metadata, and equipment record schema (identity, technical fields, audit & review fields). All owned by PhaseAlpha post-engagement.

🧠
7 Production Prompt Sets

Page classifier, table detector, crop orchestrator, extraction, normalization, mapping agent, anti-hallucination. Each prompt versioned and documented.
