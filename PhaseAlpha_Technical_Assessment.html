Overview

Executive Summary

PhaseAlpha is building a SaaS product that automates the extraction of HVAC equipment schedule data from PDF building plan sets. Today, HVAC contractors manually transcribe this data for pricing and configuration — a slow, error-prone process where the contractor owns all risk.

This document defines the complete technical architecture required to solve that problem at production quality. It is grounded in analysis of real engineering plan sets from three firms (Bluestone Engineering, MODUS Engineering, and Shive Hattery), hands-on extraction testing, and research into the relevant tool landscape.

The core finding: this is not just an extraction problem. It is a detection + rendering + extraction + trust problem. A naive single-pass approach will appear to work in demos but produce unreliable data in production. The architecture must be fail-safe, not merely high-accuracy.

Key constraint: bad data is worse than incomplete data. Pricing or estimating off fabricated values creates downstream liability risk for HVAC contractors. The system must return null for unreadable values — never infer or guess.
Strategy

Recommended Pilot Path

Before building on the existing .NET platform, the team should validate the extraction pipeline in a faster iteration environment.

Phase 0 — n8n Pilot (Weeks 1–3)

Build and test the full extraction workflow in n8n. This gives the team visual workflow control, fast iteration, and built-in HTTP/API connectors without writing backend code. All seven pipeline stages can be wired up, tested against real PDFs, and tuned before any .NET work begins.

Phase 1 — .NET Integration (Weeks 4+)

Once the n8n pilot proves the extraction logic, the .NET platform takes over as the production orchestration owner. Python microservices handle the AI-heavy stages (crop rendering, vision extraction) and are called from .NET via REST. n8n can remain as a workflow automation layer for non-critical paths.

Why n8n first: The pipeline has 7–10 distinct stages with conditional branching, fallback logic, and retry behavior. Wiring this directly into .NET before the extraction logic is proven wastes engineering time. n8n makes the workflow visible and testable in hours, not days.

Analysis of plan sets from three engineering firms surfaced several non-obvious facts that directly shape the architecture.

FirmProject TypePDF TypeComplexity

Single-Pass Extraction Fails on Complex Pages

Testing confirmed that rendering a full schedule page and sending it to a vision model in one pass produces unreliable results for dense or full-width tables. The two-pass approach — detect table regions at page scale, then crop and re-render each table at higher DPI — materially improves extraction quality.

Test result: VAV box schedule confidence improved from 0.76 to 0.91 after cropping. Full data values that were illegible at page scale became fully readable.
The two-pass crop pipeline is not optional. It is the difference between apparent accuracy in demos and real accuracy in production.

Full-Width Schedules Are a Distinct Problem Class

AHU and Rooftop Unit schedules spanning the full width of a 30"×42" sheet cannot be extracted from a single render at any practical DPI. These require a horizontal crop-and-stitch approach: split the table into left and right (or left/center/right) segments, extract each independently, and reassemble by matching column positions.

Heuristic: if a detected table spans more than 80% of the page width, trigger the horizontal split pass.
Architecture

Full System Architecture

The system requires seven distinct functional layers. Each layer has a specific job and should not be collapsed with adjacent layers.

Layer 1Document Ingestion & Routing

Accepts uploaded plan PDFs, stores originals, splits into pages, detects vector vs image content, and enqueues pages for classification. Lives in the existing .NET SaaS platform (or n8n during pilot).

  • Accept multi-page plan PDFs up to several hundred pages
  • Detect vector PDFs (text-extractable) vs image-based/raster pages
  • Create processing job with source traceability — every output row links back to the source page
  • Enqueue pages for classification — do not process everything immediately
Layer 2Page Classification

Classifies each page as schedule, legend, floor plan, detail drawing, notes, or irrelevant. Prevents expensive vision passes from being wasted on non-schedule content. A lightweight classifier is sufficient.

  • Target types: schedule, legend, drawing/plan, notes, irrelevant
  • Use title/sheet number keywords as primary signal (e.g., M6.01, MECHANICAL SCHEDULES)
  • Fall back to visual classification for pages without clear title cues
  • Route schedule pages to table detection — route everything else to archive
Layer 3Table Detection

Identifies every distinct equipment schedule table on a classified page, returning a bounding box (page percentage coordinates) and table type label for each. This is where Detector A and Detector B operate.

Detector A — LlamaParse Recommended Primary

Combines traditional text extraction with agentic LLM/VLM reasoning. Explicitly designed for complex real-world documents including messy layouts, split tables, charts, and scans.

  • Handles messy HVAC schedules well — reconstructs reading order, preserves context across pages
  • Strong on vector PDFs — starts with accurate text extraction and uses LLMs to fix layout issues
  • Production-scale: fast, scalable, used widely in AEC/RAG contexts

Detector B — Docling Fallback

Free, MIT-licensed toolkit using computer vision (DocLayNet for layout analysis + TableFormer for table structure recognition).

  • Excellent table structure recognition via TableFormer — handles cell spans, hierarchical headers, empty cells
  • Runs fully locally — self-hosted, no cloud dependency, strong data privacy story
  • Limitations: inconsistent on complex merged cells and irregular HVAC-specific formatting
Pilot strategy: run both detectors on every document and compare table counts, bounding boxes, and missed detections. In production: run Detector A first; invoke Detector B only when Detector A fails or confidence degrades.

Layer 4Rendering & Crop Orchestration

The operational heart of the system and the layer most commonly underbuilt in first-pass designs. Takes table bounding boxes from the detector and produces correctly-sized, high-DPI cropped images for the extraction model.

Required Pipeline Steps

  1. Full page render at 150 DPI — fast scan only, never used for value extraction
  2. Receive bounding boxes — convert from page-percentage coordinates to pixel coordinates
  3. Crop each table region — add 5% padding on all sides to avoid edge clipping
  4. Re-render the cropped region at target DPI (see table below)
  5. Full-width split — if detected table width exceeds 80% of page width, render left and right halves separately, extract each, stitch by matching column positions
  6. Retry on low confidence — expand crop boundary by 10% and retry once; if retry fails, route to human review
Use CaseDPIWhenNotes

Dense table crop400Smaller font, tighter columnsMODUS/Shive-Hattery style
Full-width schedule (split L/R)400AHU/RTU spanning full 30×42 sheetCrop each half, stitch results

Layer 5Extraction

Sends cropped table images to a vision model and receives raw cell-level content. This layer must operate on crops only — never on full-page images.

Model Recommendation

Claude Opus/Sonnet 3.5/4 is the recommended primary extraction model based on table comprehension capability. GPT-4o is a valid alternative for comparison testing. Azure Document Intelligence is recommended for enterprise deployments where single-cloud compliance matters.

Extraction Prompt Rules

  • Extract all visible rows and columns exactly as shown — do not infer missing values
  • Unreadable cell = null — never guess, never copy from adjacent row
  • Normalize multi-line column headers into a single string separated by ' / '
  • Prefix uncertain values with '?' for downstream flagging
  • Return structured JSON only — no markdown, no preamble
  • Include a confidence score 0.0–1.0 per table
Critical failure mode: a model can produce a suspiciously complete row from an unreadable crop by inferring plausible-looking values. The extraction prompt must explicitly prohibit this. Unreadable = null, always.

Option A — System Prompt

You are a column header mapping specialist for HVAC engineering documents. You have a knowledge base of canonical field names and their known synonyms, organized by equipment type. When given raw column headers and an equipment type, return a JSON mapping of each raw header to its canonical field name. Rules: map every header — unreadable = null, never invent canonical fields, use equipment type context to disambiguate. Flag confidence < 0.85 as needs_review: true. Return JSON only: { "mappings": [{ "raw_header": "Unit Tag", "canonical": "equipment_id", "confidence": 0.98, "needs_review": false }] }

Firm Fingerprinting Optional Enhancement

Once enough documents are processed, the system can identify the engineering firm by matching the header set against known firm profiles, loading firm-specific mappings first before falling back to the general knowledge base.

Bluestone Engineering: equipment_id → 'Symbol', cfm_max → 'CFM Cooling Max' MODUS Engineering: equipment_id → 'Unit Tag', motor_hp → 'HP' Shive Hattery: equipment_id → 'Mark', area_served → 'Service'
Layer 6bSchema Normalization

Once column headers are mapped, normalization slots values into the canonical schema — handling unit conversion, terminology normalization, and remarks concatenation. Column names are already resolved, so this layer focuses purely on value standardization.

OpenAI Structured Outputs is recommended as the primary normalization layer because it enforces the JSON schema at the model response layer, making schema violations impossible. Claude can be used as an optional second-pass repair tool on harder documents.

Layer 7Validation, Anti-Hallucination & Trust

Checks normalized records for integrity issues before they enter the review queue or export pipeline. This layer is what makes the system trustworthy rather than just accurate.

Validation Checks

  • Reject impossible values (negative CFM, temperature outside plausible range)
  • Flag blank required fields (equipment_id, area_served, cfm_max)
  • Detect duplicate equipment tags within the same document
  • Identify suspiciously complete rows extracted from low-confidence crops
  • Optional: cross-model verification — if Claude and GPT-4o disagree on a value, flag for review

Product Requirements

Review UI Requirements

The review UI is where human judgment corrects what the AI got wrong. It must be built directly into the SaaS product — not a separate admin panel.

Core Requirements

  • Side-by-side view: cropped table image on the left, extracted JSON fields on the right — without the source image next to the data, review is slow and unreliable
  • Flagged cells highlighted in the extracted view — reviewers see exactly which fields triggered a flag
  • Inline editing: reviewer edits a field directly in the UI, change saved with reviewer ID and timestamp
  • Row-level approve/reject — granular enough to pass clean rows while blocking uncertain ones
  • Bulk approval for high-confidence tables — reviewers should not touch records above a confidence threshold
  • Navigation: jump between flagged records within a document without returning to a list view

Data Privacy & Legal Framework

Data Classification

  • Raw PDF files: third-party proprietary engineering documents. PhaseAlpha does not own these. Storage should be time-limited and access-controlled.
  • Extracted schedule data: derived data. Ownership depends on contract terms with contractors. EULA must address this explicitly.
  • Model training: extracted data should not be used to train third-party models without explicit contractor consent.

Implementation

Prompt Strategy

The system requires seven distinct prompt sets. These should not be collapsed — each has a specific job and different quality requirements.

Prompt 1
Page Classifier
Classifies page type from a full-page render. Output: { page_type, confidence, reason }. Keep cheap and fast — use Haiku or equivalent.
Prompt 2
Table Detector
Identifies all schedule tables on a classified page. Output: array of { table_name, equipment_type, bbox_pct }. Must detect multiple tables per page reliably.
Prompt 3
Raw Extraction
Extracts raw cell values from a cropped table image. Hard rule: unreadable = null. No inference. No completion. This prompt is the anti-hallucination gate.
Prompt 4
Column Header Mapper
Maps variant column headers from raw extraction to canonical field names. Input: array of raw header strings. Output: { raw_header: canonical_field } mapping.
Prompt 5
Remarks Consolidator
Combines remarks column text, numbered note fragments, and row-tied callouts into a single remarks_combined string. Must handle footnote references like '1, 2, 3, 4'.
Prompt 6
Normalization
Produces normalized JSON matching the canonical schema. Uses Structured Outputs to enforce schema compliance. Never backfills unknowns — preserves null for missing values.
Prompt 7
Anomaly Validator
Checks a normalized record for integrity issues: duplicate tags, impossible values, suspicious completeness, missing required fields. Output: array of { failure_code, field, description }.

Automation Sequence

Ten automations wire the system together end to end.

A1
File Intake
Trigger: contractor uploads plan PDF. Creates job, stores original, splits pages, detects PDF type, enqueues pages for classification.
A2
Page Router
Trigger: page created. Classifies page type. Sends schedule pages to table detection. Archives non-schedule pages.
A3
Detector A with Fallback to B
Trigger: schedule page ready. Runs LlamaParse. If Detector A fails or underperforms threshold, runs Docling. Persists both results during pilot.
A4
Crop Orchestration
Trigger: bounding boxes available. Crops each table. Re-renders at appropriate DPI. Splits full-width tables. Retries with expanded crop on weak confidence.
A5
Extraction Pipeline
Trigger: crop ready. Sends crop to vision model. Returns raw cells with null for unreadable values. Stores source-to-output traceability.
A6
Normalization Pipeline
Trigger: raw extraction complete. Maps columns to canonical schema. Normalizes values. Concatenates remarks. Assigns confidence scores.
A7
Validation & Anti-Hallucination
Trigger: normalized record created. Runs business-rule checks. Detects suspicious patterns. Optional cross-model verification. Assigns failure codes.
A8
Review Queue
Trigger: low confidence, anomaly found, or detector mismatch. Presents side-by-side crop and extracted fields to reviewer. Reviewer edits and approves inline.
A9
Export to Takeoff Engine
Trigger: record approved. Writes normalized JSON. Optionally writes Excel/CSV. Attaches to project/job in .NET SaaS.
A10
Learning Loop
Trigger: reviewer edits output. Stores original extraction alongside corrected version. Tags failure type. Feeds evaluation set for prompt and rule tuning.
