Phased Build Plan
Four standard phases for AI implementation projects. Each phase has testable entry and exit criteria before the next begins.
1
Foundation
Establish all prerequisites — credentials, data access, environment setup, and approved workflow logic — before any automation is built.
Low Risk [X] weeks

Deliverables

  • All API credentials secured and tested
  • Approved workflow logic diagram
  • Data source access confirmed
  • Development environment live
  • Discovery gaps resolved

Key Activities

  • Credential collection and validation
  • Workflow mapping and approval session
  • Sandbox/staging environment setup
  • Data quality assessment
  • Resolve open items from discovery

Entry Criteria

  • SOW signed and project kicked off
  • Client champion identified
  • Tool stack and platform confirmed
  • Access requests submitted
✅ Exit Criteria (Done When…)

All credentials validated. Workflow logic approved in writing by client. Dev environment live and connection-tested. No open blockers remain.

⚠️ Risk Note

Credential delays are the most common Phase 1 blocker. Start access requests on Day 1 of kickoff — do not wait for discovery sign-off.

2
Build
Construct the core automation workflows, AI integrations, and data pipelines in the development environment against approved specs.
Medium Risk [X] weeks

Deliverables

  • Core workflows built and internally tested
  • AI/LLM integration connected and prompts tuned
  • Error handling and fallback logic implemented
  • Logging and monitoring configured
  • Build sign-off from Plus 1 team

Key Activities

  • Workflow construction in [platform]
  • Prompt engineering and iteration
  • API integration and data mapping
  • Internal QA against test cases
  • Edge case and error path testing

Entry Criteria

  • Phase 1 exit criteria met
  • All credentials in place
  • Workflow logic approved
  • Test data set ready
✅ Exit Criteria (Done When…)

All workflows execute end-to-end on test data without errors. AI outputs reviewed internally and meet accuracy threshold. Error handling tested and logged.

⚠️ Risk Note

Scope creep is highest in this phase. Any additions to the workflow spec require a change request — do not absorb undocumented requests into the build.

3
Test & UAT
Client-side user acceptance testing with real data. Validate accuracy, edge cases, and usability before production deployment.
Medium Risk [X] weeks

Deliverables

  • Completed UAT sign-off document
  • All critical bugs resolved
  • Edge cases documented and handled
  • Client approval to proceed to production

Key Activities

  • UAT session facilitation with client users
  • Real-data testing and output review
  • Bug triage and fix cycle
  • Prompt refinement based on feedback
  • Acceptance sign-off collection

Entry Criteria

  • Phase 2 exit criteria met
  • UAT participants confirmed by client
  • Real test data prepared
  • UAT schedule agreed
✅ Exit Criteria (Done When…)

Client signs UAT acceptance document. Zero critical bugs open. AI output accuracy meets agreed threshold. Client champion approves go-live.

⚠️ Risk Note

UAT requires active client participation. If client testers are unavailable, go-live slides. Confirm UAT participants and dates before Phase 2 begins.

4
Deploy & Handoff

  • 30-day monitoring plan active

Key Activities

  • Production cutover and go-live
  • User training and walkthrough
  • Documentation handoff
  • Monitoring and alerting setup
  • 30-day checkpoint scheduled

Entry Criteria

  • Phase 3 exit criteria met
  • UAT sign-off received
  • Production credentials ready
  • Training date confirmed
✅ Exit Criteria (Done When…)

System live in production with no critical issues. Training complete and attendance confirmed. All documentation delivered to client SharePoint. 30-day support window starts.

⚠️ Risk Note

Rollback plan required before go-live. Define what "rollback" means for this automation (disable trigger, revert to manual process) and who authorizes it.

Risk Register
AI-specific risks evaluated for this engagement. High-impact items are escalated immediately — not documented and ignored.
R-1 AI Output Accuracy / Hallucination Impact: High Likelihood: Medium
Description

LLM generates incorrect, fabricated, or inconsistent outputs that enter client workflows or are presented to end users without human review.

Mitigation

Define clear accuracy thresholds in UAT. Implement human-in-the-loop review for high-stakes outputs. Add confidence scoring and fallback routing.

Contingency

Revert to manual process for affected workflow. Engage in prompt re-engineering sprint. Escalate to client and document incident.

Owner

[AI Architect / Lead Builder]

R-2 Scope Creep — Undocumented Workflow Additions Impact: High Likelihood: High
Description

Client requests additional workflows, automations, or AI features mid-build that were not included in the signed SOW.

Mitigation

Reference scope document at every check-in. Anything outside scope goes through formal change request with cost/timeline impact stated before work begins.

Contingency

Pause on addition until change order is signed. Document request and defer to post-MVP phase if not approved.

Owner

[Project Manager]

R-3 API / Credential Access Delays Impact: High Likelihood: Medium
Description

Client IT does not provide API keys, OAuth access, or system credentials in time to meet build start date.

Mitigation

Submit access requests on Day 1 of kickoff. Escalate at +3 days if no response. Build with stub/mock data while access is pending.

Contingency

Adjust timeline formally. Document delay cause and impact. Do not absorb delays silently into the build schedule.

Owner

[PM / Client IT]

R-4 Data Quality Issues Impact: Medium Likelihood: High
Description

Source data is incomplete, inconsistently formatted, or contains errors that degrade AI output quality or cause workflow failures.

Mitigation

Conduct data quality assessment in Phase 1. Define minimum data quality standards. Client is responsible for data cleaning unless contracted otherwise.

Contingency

Add data validation layer to workflow. Flag bad records for human review rather than failing silently. Escalate cleaning requirements to client.

Owner

[AI Architect / Client]

R-5 Low User Adoption / Change Resistance Impact: Medium Likelihood: Medium
Description

End users resist or avoid the new AI workflow, reverting to manual processes and negating ROI of the implementation.

Mitigation

Involve end users in UAT. Provide training with real use cases. Ensure client champion is visibly endorsing the system. Document the "why" for users.

Contingency

Schedule additional training sessions. Identify and address specific objections. Escalate to client leadership if adoption remains low at 30-day checkpoint.

Owner

[Client Champion / PM]

R-6 Third-Party Model / Platform Pricing or Availability Change Impact: Medium Likelihood: Low
Description

Model provider (OpenAI, Anthropic, etc.) changes pricing, deprecates a model version, or experiences an outage that affects the build.

Mitigation

Pin model versions where possible. Design prompt architecture to be model-agnostic. Monitor provider announcements throughout build.

Contingency

Switch to alternative model or provider. Communicate impact and updated timeline to client. Track model costs throughout project.

[AI Architect]
